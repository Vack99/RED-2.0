## Context

Today Forge is a single-operator admin app. There is exactly one human who logs in â€” the operator â€” and **`clientes` are operator-created CRM rows that never authenticate** (`src/lib/data/clientes.ts`). The Agenda roster reads each member's package balance straight off those rows (*"8 clases Â· quedan 3"*, *"Ilimitado"*, *"por renovar"*). RLS today is keyed to `(select auth.uid())` as the lone operator ([ADR-0001](0001-supabase-rls-no-orm.md)).

The RED client mock breaks that single-population assumption. It is a **member-facing** app where people who train at the gym **self-register** (name, email+password, phone `+52` for WhatsApp, a required terms+privacy checkbox), log in, and see their own *paquete*, *reservas*, and *asistencia*. So the platform now has **two kinds of human** against one shared database: the gym's **operator/owner** (authors the catalog, runs *Pasar lista*, sells *paquetes*) and the gym's **members** (book *clases*, view their own data). These are different populations with different powers, different screens, and different write surfaces â€” but they share one gym's data and one `auth.users` table (the platform is one shared Supabase project, [ADR-0008](0008-platform-multitenant-gym-rls-brand-modules.md)).

Two failure modes must be designed out from the start. **(1)** If operator and member were the same auth role / same profile row, a member could touch operator-only catalog or another member's record â€” the RLS boundary would have nothing to discriminate on (data-model Â§5.7). **(2)** If self-registration always minted a fresh member record, the person the operator already knows as a `clientes` row â€” with a live *paquete* balance and attendance history â€” would split into a duplicate, **orphaning the `paquetes` balance** the operator was tracking (data-model Â§2). This ADR formalises the locked identity model (data-model Â§1â€“Â§3) that prevents both. It does **not** cover platform infrastructure ([ADR-0008](0008-platform-multitenant-gym-rls-brand-modules.md)) or the scheduling/booking schema ([ADR-0010](0010-class-scheduling-absolute-starts-derived-occupancy.md)); it is the identity spine those consume.

## Decision

**Identity is two-tier, operator and member are distinct roles in distinct profile tables joined by one membership map, and a self-registering person claims their pre-existing `member` row by email/phone match.** Concretely, against the data-model Â§4 schema:

### Two-tier authentication

- **Members self-register.** Per the RED client mock, `registro` collects `full_name`, `email` + `password`, `phone_e164` (`+52`, used for WhatsApp), and a **required terms + privacy checkbox**, stamping `terms_accepted_at` / `privacy_accepted_at` on the `member` row (Â§4). **There is no gym selector in the UI** â€” the gym is resolved from the **domain the person registers on** (host â†’ `gym`, the same hostâ†’tenant resolution [ADR-0008](0008-platform-multitenant-gym-rls-brand-modules.md) owns). The resolved `gym_id` is an authoritative server-side fact, **not** a client-supplied field.
- **Gyms / owners are invited / provisioned manually.** A new gym ("Forge", "RED", future) is stood up out-of-band: the owner `auth.users` account, the `gym` row, and the domain are provisioned by the platform operator. **Self-serve gym onboarding is deferred / future** (data-model Â§6 parked â€” *"out of scope now; gyms invited/provisioned"*); this ADR does not design it.
- **Login (`entrar`) is email + password only**, with forgot-password. **No phone-OTP, no social login** (data-model Â§1). Authorization in server code uses `getClaims()` / `getUser()`, **never** `getSession()` ([ADR-0001](0001-supabase-rls-no-orm.md), Â§5.8).

### Operator â‰  member â€” distinct tables, one membership map

- Operator and member are **never the same row or the same auth role** (data-model Â§5.7). They live in **distinct profile tables**: a **thin operator profile** (what remains of the old `perfil` once brand/location/contact graduate to `gym`, Â§4) versus the **`member`** profile (Â§4). Brand-facing identity lives on `gym`; the operator profile is the slimmed-down staff record.
- One shared table maps identity â†’ tenant + role: **`gym_membership (user_id â†’ auth.users, gym_id â†’ gym, role âˆˆ owner|operator|member)`** (Â§4). This single row is what **every RLS predicate resolves "which gym + what role" from**. A given `auth.users` row has a `gym_membership` for each gym it belongs to; the role on that row decides operator-class vs member-class access.

### `member` evolves from `clientes`, with a nullable `auth_user_id`

- There is **one `member` table per gym**, evolving the existing operator-created `clientes` CRM rows (Â§4 evolution map: `clientes â†’ member`). It carries `gym_id`, `full_name`, `email`, `phone_e164`, `member_since`, `favorite_class_type_id`, `notifications_enabled`, `terms_accepted_at`, `privacy_accepted_at`, and â€” the load-bearing column â€” **`auth_user_id NULL`** (Â§4, Â§2).
- **`auth_user_id` is nullable on purpose, and stays nullable.** An operator-created member who has not yet registered is a real `member` row with `auth_user_id = NULL`; for a member who never self-registers it stays `NULL` **forever**. It becomes non-null only when a person registers and claims the row. This is what lets the operator keep running the CRM exactly as today while members trickle into self-service.

### Claim-by-match (data-model Â§2 â€” the locked default)

When a person self-registers, the platform **matches on email/phone against pre-existing `member` rows in that gym** before creating anything:

- **Match â†’ claim.** Populate `auth_user_id` on the matched `member` row with the new `auth.uid()`. The person **claims** the operator-authored record; their *paquete* balance (`subscription` state, Â§4) and attendance history **carry over** untouched. A `gym_membership(user_id, gym_id, role='member')` row is written for them.
- **No match â†’ create.** Mint a **new `member` row** for that gym with `auth_user_id` set to the new `auth.uid()`, plus the matching `gym_membership` row.

Claim-by-match is the **locked default** precisely because it prevents the two Â§2 failure modes: duplicate members and orphaned `paquetes` balances. The data-model records a **veto point** â€” *if the operator would rather always create fresh rows, that must be raised before Phase 3* â€” but absent that veto, claim-by-match stands (see Consequences).

### The RLS backbone â€” read/write matrix (data-model Â§3)

Every tenant table is `gym_id`-scoped with RLS ([ADR-0008](0008-platform-multitenant-gym-rls-brand-modules.md); new `public` tables auto-enable RLS via the `rls_auto_enable` event trigger, Â§5.2). Policies discriminate by **role resolved from `gym_membership`**. Member-owned reads are doubly scoped â€” the owning member **and** the gym's operator â€” never another member.

### The isolation invariant (data-model Â§5.6)

**Tenant isolation is enforced by RLS keyed to `gym_membership` + role â€” never by trusting the `proxy.ts` tenant header.** The hostâ†’gym resolution that `proxy.ts` performs is **brand/UX only** ([ADR-0008](0008-platform-multitenant-gym-rls-brand-modules.md)); it selects which brand renders and seeds the registration gym, but it is **not** an authorization fact. Even a forged or mismatched header cannot cross a gym boundary, because the `WHERE` clause every policy compiles to is `gym_id = (the caller's gym_membership gym)`, evaluated in Postgres under the caller's own JWT. This is the same posture as [ADR-0001](0001-supabase-rls-no-orm.md): RLS is the boundary, the app server is not.

## Consequences

- **Two populations, one database, one auth table â€” cleanly separated.** Because role lives on `gym_membership` and profiles are split (thin operator profile vs `member`), the same RLS engine serves operator and member within a gym with no ambiguity: the policy reads the role and grants the right surface. A member can never reach operator-only catalog writes or another member's transactional rows; the discrimination is structural, not conventional.
- **The CRM keeps working unchanged through the transition.** A `member` row with `auth_user_id = NULL` is a fully valid operator-authored record. Operators continue creating members, selling *paquetes*, and running *Pasar lista* exactly as today; self-service adoption is incremental, not a cutover. The operator's secondary-write grant on member-owned tables is what preserves walk-ins and back-dated attendance on the new schema.
- **Claim-by-match carries balances; it is the default, and the veto is bounded.** When a known *cliente* registers, their *paquete*/`subscription` balance and history follow them â€” no duplicate, no orphan (Â§2). The recorded **veto point** (*always-fresh-rows, if the operator prefers â€” raise before Phase 3*) is a **deliberately time-boxed escape hatch, not an open question**: absent an explicit objection before the Phase-3 tenant/identity work, claim-by-match is built. A future reader must not silently swap to always-create â€” that reopens the duplicate-member / orphaned-`paquetes` gap this decision closes, and would strand balances the operator was tracking.
- **Match key precision is the one sharp edge.** Matching on email/phone assumes the operator-entered `email`/`phone_e164` and the self-registered values reconcile. Phone is normalised to E.164 (`phone_e164`) to make the comparison deterministic; a mismatch (typo, second number) degrades safely to *No match â†’ create*, i.e. a duplicate the operator can later reconcile â€” never a wrong-person claim. Email/phone are treated as the identity key for the claim, consistent with phone being effectively required (CONTEXT.md's flagged tension: WhatsApp retention makes phone load-bearing).
- **Registration writes are server-authoritative on gym.** Because there is no gym selector and the gym comes from the host, `gym_id` on a new `member`/`gym_membership` is set server-side from the resolved tenant â€” never from client input. A future reader must not add a client-supplied gym field to `registro`; it would be an authorization input, which Â§5.6 forbids.
- **Deferred surfaces are named, not forgotten.** Self-serve gym onboarding (owners provision themselves) and real payment processing are **parked with stated defaults** (data-model Â§6): gyms are invited/provisioned manually, and `membresÃ­a`/checkout is **UI-only for v1** â€” `plan` selection and `subscription` status are built, charging is deferred. This ADR scopes identity only; it does not authorise either.
- **This is the identity foundation [ADR-0010](0010-class-scheduling-absolute-starts-derived-occupancy.md) consumes.** Booking a *clase*, decrementing the member's *paquete*, and *Pasar lista* all resolve "who is this, in which gym, with what role" from `gym_membership` + the claimed `member` row defined here. Scheduling's correctness rules (occupancy derived, `Ilimitado` exempt from consumption, no-show still consumes) live in that ADR and ride on this identity, not in this one.
- **No code, no migration is authorised by this ADR.** Per the roadmap sequencing principle, the tenant/identity spine lands in Phase 3 as expand/contract migrations. One column is explicitly **not** subject to the tightening half of that pattern: **`auth_user_id` is added nullable and stays nullable permanently** â€” never backfilled, never enforced `NOT NULL` â€” because an unregistered CRM member has no `auth.users` row to point at, and the "CRM keeps working unchanged" guarantee depends on its permanent nullability. The expand/contract `add nullable â†’ backfill â†’ enforce` technique applies only to genuinely-tightened columns (e.g. `gym_id` on the evolved `member` rows, which every row must carry), not to `auth_user_id`. Policy cutover is itself expand/contract (add gym-scoped policies â†’ cut over â†’ drop the per-`auth.uid()` ones), gated by the cross-gym denial test written **before** the policy it guards. This document records the decision; it does not perform the work.
