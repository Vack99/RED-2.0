# ADR-0009 ‚Äî Identity & member model: two-tier auth, operator‚âÝmember roles, claim-by-match

**Status:** Accepted ¬∑ **Date:** 2026-06-29 ¬∑ **Amended:** 2026-07-02 (claim mechanics ‚Äî verified-email-gated claim, atomic RPC), 2026-07-05 (open-enrollment accepted debt, I1; see Amendments below), 2026-07-15 (invited-member activation door ‚Äî admin-provisioned pre-confirmed account + set-password) ¬∑ **Builds on:** [ADR-0001](0001-supabase-rls-no-orm.md) (RLS-as-boundary, no ORM, `getClaims()`/`getUser()`), [ADR-0008](0008-platform-multitenant-gym-rls-brand-modules.md) (one shared Supabase + per-gym tenant model ¬∑ *isolation is never the `proxy.ts` header*)

## Context

Today Forge is a single-operator admin app. There is exactly one human who logs in ‚Äî the operator ‚Äî and **`clientes` are operator-created CRM rows that never authenticate** (`src/lib/data/clientes.ts`). The Agenda roster reads each member's package balance straight off those rows (*"8 clases ¬∑ quedan 3"*, *"Ilimitado"*, *"por renovar"*). RLS today is keyed to `(select auth.uid())` as the lone operator ([ADR-0001](0001-supabase-rls-no-orm.md)).

The RED client mock breaks that single-population assumption. It is a **member-facing** app where people who train at the gym **self-register** (name, email+password, phone `+52` for WhatsApp, a required terms+privacy checkbox), log in, and see their own *paquete*, *reservas*, and *asistencia*. So the platform now has **two kinds of human** against one shared database: the gym's **operator/owner** (authors the catalog, runs *Pasar lista*, sells *paquetes*) and the gym's **members** (book *clases*, view their own data). These are different populations with different powers, different screens, and different write surfaces ‚Äî but they share one gym's data and one `auth.users` table (the platform is one shared Supabase project, [ADR-0008](0008-platform-multitenant-gym-rls-brand-modules.md)).

Two failure modes must be designed out from the start. **(1)** If operator and member were the same auth role / same profile row, a member could touch operator-only catalog or another member's record ‚Äî the RLS boundary would have nothing to discriminate on (data-model ¬ß5.7). **(2)** If self-registration always minted a fresh member record, the person the operator already knows as a `clientes` row ‚Äî with a live *paquete* balance and attendance history ‚Äî would split into a duplicate, **orphaning the `paquetes` balance** the operator was tracking (data-model ¬ß2). This ADR formalises the locked identity model (data-model ¬ß1‚Äì¬ß3) that prevents both. It does **not** cover platform infrastructure ([ADR-0008](0008-platform-multitenant-gym-rls-brand-modules.md)) or the scheduling/booking schema ([ADR-0010](0010-class-scheduling-absolute-starts-derived-occupancy.md)); it is the identity spine those consume.

## Decision

**Identity is two-tier, operator and member are distinct roles in distinct profile tables joined by one membership map, and a self-registering person claims their pre-existing `member` row by email/phone match.** Concretely, against the data-model ¬ß4 schema:

### Two-tier authentication

- **Members self-register.** Per the RED client mock, `registro` collects `full_name`, `email` + `password`, `phone_e164` (`+52`, used for WhatsApp), and a **required terms + privacy checkbox**, stamping `terms_accepted_at` / `privacy_accepted_at` on the `member` row (¬ß4). **There is no gym selector in the UI** ‚Äî the gym is resolved from the **domain the person registers on** (host ‚Üí `gym`, the same host‚Üítenant resolution [ADR-0008](0008-platform-multitenant-gym-rls-brand-modules.md) owns). The resolved `gym_id` is an authoritative server-side fact, **not** a client-supplied field.
- **Gyms / owners are invited / provisioned manually.** A new gym ("Forge", "RED", future) is stood up out-of-band: the owner `auth.users` account, the `gym` row, and the domain are provisioned by the platform operator. **Self-serve gym onboarding is deferred / future** (data-model ¬ß6 parked ‚Äî *"out of scope now; gyms invited/provisioned"*); this ADR does not design it.
- **Login (`entrar`) is email + password only**, with forgot-password. **No phone-OTP, no social login** (data-model ¬ß1). Authorization in server code uses `getClaims()` / `getUser()`, **never** `getSession()` ([ADR-0001](0001-supabase-rls-no-orm.md), ¬ß5.8).

### Operator ‚âÝ member ‚Äî distinct tables, one membership map

- Operator and member are **never the same row or the same auth role** (data-model ¬ß5.7). They live in **distinct profile tables**: a **thin operator profile** (what remains of the old `perfil` once brand/location/contact graduate to `gym`, ¬ß4) versus the **`member`** profile (¬ß4). Brand-facing identity lives on `gym`; the operator profile is the slimmed-down staff record.
- One shared table maps identity ‚Üí tenant + role: **`gym_membership (user_id ‚Üí auth.users, gym_id ‚Üí gym, role ‚àà owner|operator|member)`** (¬ß4). This single row is what **every RLS predicate resolves "which gym + what role" from**. A given `auth.users` row has a `gym_membership` for each gym it belongs to; the role on that row decides operator-class vs member-class access.

### `member` evolves from `clientes`, with a nullable `auth_user_id`

- There is **one `member` table per gym**, evolving the existing operator-created `clientes` CRM rows (¬ß4 evolution map: `clientes ‚Üí member`). It carries `gym_id`, `full_name`, `email`, `phone_e164`, `member_since`, `favorite_class_type_id`, `notifications_enabled`, `terms_accepted_at`, `privacy_accepted_at`, and ‚Äî the load-bearing column ‚Äî **`auth_user_id NULL`** (¬ß4, ¬ß2).
- **`auth_user_id` is nullable on purpose, and stays nullable.** An operator-created member who has not yet registered is a real `member` row with `auth_user_id = NULL`; for a member who never self-registers it stays `NULL` **forever**. It becomes non-null only when a person registers and claims the row. This is what lets the operator keep running the CRM exactly as today while members trickle into self-service.

### Claim-by-match (data-model ¬ß2 ‚Äî the locked default)

When a person self-registers, the platform **matches on email/phone against pre-existing `member` rows in that gym** before creating anything:

- **Match ‚Üí claim.** Populate `auth_user_id` on the matched `member` row with the new `auth.uid()`. The person **claims** the operator-authored record; their *paquete* balance (`subscription` state, ¬ß4) and attendance history **carry over** untouched. A `gym_membership(user_id, gym_id, role='member')` row is written for them.
- **No match ‚Üí create.** Mint a **new `member` row** for that gym with `auth_user_id` set to the new `auth.uid()`, plus the matching `gym_membership` row.

Claim-by-match is the **locked default** precisely because it prevents the two ¬ß2 failure modes: duplicate members and orphaned `paquetes` balances. The data-model records a **veto point** ‚Äî *if the operator would rather always create fresh rows, that must be raised before Phase 3* ‚Äî but absent that veto, claim-by-match stands (see Consequences).

### The RLS backbone ‚Äî read/write matrix (data-model ¬ß3)

Every tenant table is `gym_id`-scoped with RLS ([ADR-0008](0008-platform-multitenant-gym-rls-brand-modules.md); new `public` tables auto-enable RLS via the `rls_auto_enable` event trigger, ¬ß5.2). Policies discriminate by **role resolved from `gym_membership`**. The matrix below classifies tables by **RLS role-class only** ‚Äî the *schemas* of the catalog/scheduling tables it names (`coach`, `class_type`, `plan`/`plan_feature`, `class_session`/`schedule_template`, `room`‚Ä¶) are owned by [ADR-0008](0008-platform-multitenant-gym-rls-brand-modules.md) / [ADR-0010](0010-class-scheduling-absolute-starts-derived-occupancy.md), not defined here; read no schema ownership into the table list. There are three classes:

| Class | Tables (¬ß4) | Write | Read |
|---|---|---|---|
| **Curated / showcased** (operator authors, members view) | `gym`, `gym_hours`, gym content (`about_value`, `facility`, `stat`, `faq`), `coach`, `class_type`(+children), `plan`(+`plan_feature`), `class_session`(+`schedule_template`), `room` | `owner`/`operator` of that gym | members of that gym (+ anon/public for marketing pages) |
| **Member-owned / transactional** (member creates own) | `member` *(self fields)*, `subscription`, `reservation`, `payment`, member stats, notification prefs, `waitlist_entry` | the member (`auth_user_id = (select auth.uid())`) ‚Äî **operator may also write** (walk-ins, *Pasar lista*) | the owning member + the gym's operator |
| **Public intake** | `contact_message` | **anon** (the public contact form) | operator |

The operator's secondary-write grant on the member-owned class is what keeps the *existing* admin flows alive on the new schema: an operator records a walk-in *reserva* or runs *Pasar lista* against a member who may never have logged in (`auth_user_id = NULL`). Member-owned reads are doubly scoped ‚Äî the owning member **and** the gym's operator ‚Äî never another member.

### The isolation invariant (data-model ¬ß5.6)

**Tenant isolation is enforced by RLS keyed to `gym_membership` + role ‚Äî never by trusting the `proxy.ts` tenant header.** The host‚Üígym resolution that `proxy.ts` performs is **brand/UX only** ([ADR-0008](0008-platform-multitenant-gym-rls-brand-modules.md)); it selects which brand renders and seeds the registration gym, but it is **not** an authorization fact. Even a forged or mismatched header cannot cross a gym boundary, because the `WHERE` clause every policy compiles to is `gym_id = (the caller's gym_membership gym)`, evaluated in Postgres under the caller's own JWT. This is the same posture as [ADR-0001](0001-supabase-rls-no-orm.md): RLS is the boundary, the app server is not.

## Consequences

- **Two populations, one database, one auth table ‚Äî cleanly separated.** Because role lives on `gym_membership` and profiles are split (thin operator profile vs `member`), the same RLS engine serves operator and member within a gym with no ambiguity: the policy reads the role and grants the right surface. A member can never reach operator-only catalog writes or another member's transactional rows; the discrimination is structural, not conventional.
- **The CRM keeps working unchanged through the transition.** A `member` row with `auth_user_id = NULL` is a fully valid operator-authored record. Operators continue creating members, selling *paquetes*, and running *Pasar lista* exactly as today; self-service adoption is incremental, not a cutover. The operator's secondary-write grant on member-owned tables is what preserves walk-ins and back-dated attendance on the new schema.
- **Claim-by-match carries balances; it is the default, and the veto is bounded.** When a known *cliente* registers, their *paquete*/`subscription` balance and history follow them ‚Äî no duplicate, no orphan (¬ß2). The recorded **veto point** (*always-fresh-rows, if the operator prefers ‚Äî raise before Phase 3*) is a **deliberately time-boxed escape hatch, not an open question**: absent an explicit objection before the Phase-3 tenant/identity work, claim-by-match is built. A future reader must not silently swap to always-create ‚Äî that reopens the duplicate-member / orphaned-`paquetes` gap this decision closes, and would strand balances the operator was tracking.
- **Match key precision is the one sharp edge.** Matching on email/phone assumes the operator-entered `email`/`phone_e164` and the self-registered values reconcile. Phone is normalised to E.164 (`phone_e164`) to make the comparison deterministic; a mismatch (typo, second number) degrades safely to *No match ‚Üí create*, i.e. a duplicate the operator can later reconcile ‚Äî never a wrong-person claim. Email/phone are treated as the identity key for the claim, consistent with phone being effectively required (CONTEXT.md's flagged tension: WhatsApp retention makes phone load-bearing).
- **Registration writes are server-authoritative on gym.** Because there is no gym selector and the gym comes from the host, `gym_id` on a new `member`/`gym_membership` is set server-side from the resolved tenant ‚Äî never from client input. A future reader must not add a client-supplied gym field to `registro`; it would be an authorization input, which ¬ß5.6 forbids.
- **Deferred surfaces are named, not forgotten.** Self-serve gym onboarding (owners provision themselves) and real payment processing are **parked with stated defaults** (data-model ¬ß6): gyms are invited/provisioned manually, and `membres√≠a`/checkout is **UI-only for v1** ‚Äî `plan` selection and `subscription` status are built, charging is deferred. This ADR scopes identity only; it does not authorise either.
- **This is the identity foundation [ADR-0010](0010-class-scheduling-absolute-starts-derived-occupancy.md) consumes.** Booking a *clase*, decrementing the member's *paquete*, and *Pasar lista* all resolve "who is this, in which gym, with what role" from `gym_membership` + the claimed `member` row defined here. Scheduling's correctness rules (occupancy derived, `Ilimitado` exempt from consumption, no-show still consumes) live in that ADR and ride on this identity, not in this one.
- **No code, no migration is authorised by this ADR.** Per the roadmap sequencing principle, the tenant/identity spine lands in Phase 3 as expand/contract migrations. One column is explicitly **not** subject to the tightening half of that pattern: **`auth_user_id` is added nullable and stays nullable permanently** ‚Äî never backfilled, never enforced `NOT NULL` ‚Äî because an unregistered CRM member has no `auth.users` row to point at, and the "CRM keeps working unchanged" guarantee depends on its permanent nullability. The expand/contract `add nullable ‚Üí backfill ‚Üí enforce` technique applies only to genuinely-tightened columns (e.g. `gym_id` on the evolved `member` rows, which every row must carry), not to `auth_user_id`. Policy cutover is itself expand/contract (add gym-scoped policies ‚Üí cut over ‚Üí drop the per-`auth.uid()` ones), gated by the cross-gym denial test written **before** the policy it guards. This document records the decision; it does not perform the work.

## Amendment ‚Äî 2026-07-02 (Phase-3 grill: claim mechanics locked)

- **The veto is closed.** The bounded veto point above was put to the product owner at the Phase-3 grill checkpoint and declined: claim-by-match stands, now beyond veto.
- **A claim executes only on a verified-email match; phone is demoted from claim key.** The original "match on email/phone" treated both keys as equal, but only email is *verified* (Supabase confirmation mail) ‚Äî phone is self-asserted at `registro`, so a phone-only claim would let anyone who knows a member's phone number register with their own email and claim that member's row, balance, and history: a wrong-person claim, the one failure ¬ß2's "degrades safely" promise forbids. Phone remains a *reconciliation hint* for the operator; phone-only matches degrade to **No match ‚Üí create**.
- **Ambiguous match ‚Üí create.** More than one unclaimed row matching the verified email in the gym: never guess ‚Äî mint a fresh row; the operator reconciles duplicates.
- **Mechanism: one atomic SECURITY DEFINER RPC.** Match ‚Üí set `auth_user_id` ‚Üí insert `gym_membership(role='member')` commit in a single transaction (no half-registered state RLS can't classify), invoked from the post-verification server action. SECURITY DEFINER is structural, not convenience: the registrant cannot see the unclaimed `member` row under RLS (they are not yet a member of anything). House posture applies (`search_path=''`, EXECUTE revoked from `public`/`anon`, granted to `authenticated` ‚Äî [ADR-0013](0013-gym-scoped-rls-mechanism.md)); the RPC re-checks `email_confirmed_at IS NOT NULL` as defense-in-depth even though confirm-email-required means no session exists before verification.

## Amendment ‚Äî 2026-07-05 (issue #28 cutover: open-enrollment is accepted debt, I1)

- **Open enrollment is accepted, and it is a SECURITY DEFINER WRITE vector.** `reclamar_o_crear_cliente` grants EXECUTE to every `authenticated` session and takes `p_gym_id` as an argument, so any verified-email user can mint a `clientes` row **and** a `gym_membership(role='member')` row into **any** `gym_id` ‚Äî not just the one their host resolves. It is trivially scriptable: `gym` rows are anon-readable (marketing pages need it), and email confirmation is global to the auth project, so one confirmed account drives the RPC against every gym id it can read. The realistic damage is **CRM pollution** ‚Äî junk member rows an operator must prune.
- **Bounded.** The vector writes only its own `member` + membership rows: **no member PII is exposed** (the attacker reads nothing back ‚Äî `clientes_member_select` scopes to their own `auth_user_id`), **no `cobro`/CLABE**, **no cross-member reads**, and **no writes to any other tenant-table surface** (the RLS write gate is `is_staff_of`, which the attacker never satisfies). It cannot escalate past minting its own rows.
- **Accepted debt, with an un-defer trigger.** Accepted while Forge is the sole live gym. Re-open when **either** a second gym goes live **or** the first abuse report lands.
- **The real mitigation is not a `p_gym_id` guard.** When triggered, gate enrollment on a server-injected gym proof: an invitation/allowlist row keyed to the gym, or a service-role-signed gym token the server action mints and the RPC verifies. A naive "check `p_gym_id` matches the caller" guard is a **non-fix** ‚Äî the RPC stays directly callable with the same JWT, so the caller just passes whichever gym id they targeted.
- **Flag (separate surface): `plantillas_member_select` leaks operator WhatsApp templates to members.** The member-read policy on `plantillas` exposes the operator's internal message templates to every member of the gym ‚Äî an operator-authoring surface, not member content. Revisit when member-facing scope is next touched.
- **Note:** at the #28 column contract, `crear_plantilla`'s 4-template cap becomes **per-gym** (`where gym_id = v_gym`). Inert today (one operator per gym); it diverges only when a gym has a second operator.

## Amendment ‚Äî 2026-07-15 (single-email activation: the invited-member door)

Locked in [PRD #130](https://github.com/Vack99/RED-2.0/issues/130); architecture in [#126](https://github.com/Vack99/RED-2.0/issues/126). This adds a **second entry door** for members the operator already registered; the self-registration door above is untouched.

- **An invited member activates against a pre-confirmed account, not the self-registration form.** Today the invitation link dropped the member into full account creation, which fired a *second* confirmation email ‚Äî a second lap to the inbox where members were lost to spam. Instead, the invitation email now lands on the activation door (`/activar`); the member confirms the email their gym registered them with (case-insensitive match against the roster row) and passes a Turnstile check, and the platform provisions the `auth.users` account server-side with **the email marked confirmed at creation**. The invitation mail delivered to that inbox *is* the verification ‚Äî the confirmation-mail step the self-registration rail relies on is redundant here and is dropped. An email that already has an account (a member of a second gym) is *not* provisioned by the activation and takes the inbox-proof path below instead.
- **Server-consumed activation tokens are minted ONLY for accounts the activation itself provisions; a pre-existing account always requires inbox proof.** The provisioning rail hands the client-app server a recovery token it consumes to establish a session *in the same request* ‚Äî so minting one for an account the activation did not create would hand a live session to that account with no proof the activator controls its inbox (a hostile gym-B operator could register a roster row with a gym-A member's email and take over the account). The edge function therefore returns `cuenta_existente` (no token) when the email already has an account, and the activation action falls to the existing recovery rail (`solicitarReset` ‚Üí `/auth/confirm?codigo=‚Ä¶&next=/restablecer`): the emailed link proves the inbox, claims this gym's membership by code on the verified recovery session, then lands on set-password. Residual accepted risk: a staff-planted roster row can trigger an *unsolicited* link-and-membership-claim email to an existing member ‚Äî but it exposes no data and hands the attacker no session, so it is accepted (the same posture as any operator-typed contact detail).
- **Set-password on a verified session, then claim.** The provisioning step establishes a session in the same request; the member sets a password (registered email shown read-only) and accepts the same terms/privacy gate as self-registration, and only *then* does the existing claim-by-code RPC run. Claim-after-password means an abandoned activation leaves the claim code intact and the link re-usable; a member who sets a password and forgets it uses the standard recovery door. This composes [ADR-0015](0015-invite-token-claim.md)'s claim-by-code with this ADR's two-tier model ‚Äî no new identity rule, no schema change.
- **The privileged auth-admin capability stays out of the apps.** Provisioning (`createUser` with email pre-confirmed) and the recovery-link mint run inside a Supabase Edge Function whose service-role key lives only in the function's environment; the client app calls it behind an HMAC `firma` and never holds the admin key. The repo's no-service-role-import property (owner ruling 2026-07-13) is preserved by placement.
- **The self-registration door is unchanged.** Walk-ins without an invitation keep hitting `registro` ‚Üí email-verified claim-by-match exactly as above; open-enrollment (I1) and its accepted debt are untouched. The activation door is additive.
