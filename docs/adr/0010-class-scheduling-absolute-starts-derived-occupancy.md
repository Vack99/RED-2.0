# ADR-0010 — Class scheduling: absolute `starts_at` + `schedule_template`, multi-coach join, derived occupancy, plan-consuming booking

**Status:** Accepted · **Date:** 2026-06-29 · **Builds on:** [ADR-0001](0001-supabase-rls-no-orm.md) (RLS-as-boundary, `server-only` DAL, no ORM), [ADR-0002](0002-derived-not-stored.md) (the derived-not-stored principle — occupancy extends it here), [ADR-0004](0004-saldo-stored-running-balance.md) (the consume decrement is a stored running balance), [ADR-0005](0005-atomic-write-rpcs.md) (the consume rides the atomic write seam) · **Rides on** [ADR-0008](0008-platform-multitenant-gym-rls-brand-modules.md) (gym-scoped RLS, one shared Supabase) and [ADR-0009](0009-identity-two-tier-auth-member-claim.md) (`member` / identity, claim-by-match) · **Scope-bounded by** the data-model doc §4 (catalog + transactional), stating §5 invariants principally 1, 3, 4, 5 — and honoring 2 (`gym_id` + RLS) and 6 (RLS-by-membership, never the `proxy.ts` header)

## Context

The admin **Agenda** mock (operator class scheduling, *Agenda Week View*) and the client booking mock (*reservar → clase → confirmada → mis reservas*) both render classes, occupancy, coaches, and a member's package balance. Both are read-only HTML; both encode shortcuts that are safe for a static mock and **wrong for a live, multi-tenant DB**:

- The Agenda lays classes out on a **weekday grid with string times** ("Lunes 07:00"). That is a presentation artifact, not a storable fact — it cannot express a one-off *clase especial*, a holiday move, or a daylight shift, and it has no absolute instant to sort, gate ("ya empezó"), or join attendance against.
- Each class card shows a single coach name. The roster shows **occupancy as a stored number** the mock decrements in place on book (`spots--`). The booking mock reduces a "quedan 3" badge by hand and never touches the member's plan.

The live Forge app already owns the money-path half of this: a paquete grants `clases_restantes` (NULL = ilimitado) as a **stored running balance** ([ADR-0004](0004-saldo-stored-running-balance.md)), and attendance consumes it atomically through `toggle_pase` / the `consumirClase` rule on the single SECURITY INVOKER write seam ([ADR-0005](0005-atomic-write-rpcs.md)). The class-scheduling model must extend that spine — gym-scoped per [ADR-0008](0008-platform-multitenant-gym-rls-brand-modules.md), member-aware per [ADR-0009](0009-identity-two-tier-auth-member-claim.md) — **without reproducing a single one of the mock's shortcuts.** This ADR formalizes the catalog/transactional shape (data-model §4) and states the §5 invariants *as* the decision. It does not re-decide tenancy ([ADR-0008](0008-platform-multitenant-gym-rls-brand-modules.md)) or identity ([ADR-0009](0009-identity-two-tier-auth-member-claim.md)); it consumes both.

## Decision

### 1. Class time is an absolute instant; recurrence is a separate template (§5.3)

A scheduled class is **`class_session`** with `starts_at timestamptz` as the canonical time — an **absolute instant**, never weekday + string. Alongside it: `gym_id`, `class_type_id`, `duration_min` (30/45/60/75/90), `capacity` (4–40), `is_special` / `special_name` (the one-off *clase especial*), `room_id?` (nullable — single room per gym is the §6 default), and `template_id?` (provenance back to the generator). The Agenda's weekday grid is **rendered from `starts_at`**, not stored as it.

Recurrence lives in **`schedule_template`** (`gym_id`, `class_type_id`, `weekday 0–5`, `start_time`, `duration_min`, `capacity`, default coach(es), `is_active`). It is the editor's *"create across N weekdays"* generator: it **materializes concrete `class_session` rows** (each carrying its own absolute `starts_at`); it is not consulted at read time to compute occupancy or attendance. Those rows are **independent once written** — so a template edit cannot reach sessions already on the calendar; it governs only sessions materialized after it. That is the same future-only-grant posture [ADR-0007](0007-editable-clases-derived-nombre-single-favorite.md) takes for the paquete catalog, and here it is a *consequence* of the materialize-to-independent-rows model, not a separate rule. A *clase especial* is a `class_session` with no `template_id` and `is_special = true`.

### 2. Coaches are a join table — never a column (§5.4)

A session's coaches are **`class_session_coach (session_id, coach_id)`**, a join table. There is **no single coach column.** `coach` is the gym-scoped curated catalog (`gym_id`, `name`, `initials`, `role`, `specialty`, `bio`, `is_active`, `sort_order`) — operator-authored, member-viewed (curated/showcased RLS class, §3). `schedule_template`'s default coach(es) are likewise multi-valued and seed the join rows when a session materializes.

### 3. Occupancy is DERIVED — never stored (§5.1, extends [ADR-0002](0002-derived-not-stored.md))

Spots are **always computed**: `spots = capacity − count(active reservations)`. `class_session` stores `capacity`; it **does not** store `spots`, `occupancy`, or `quedan`. The mocks' in-place `spots--` is **mock-only** and must not be reproduced — it is precisely the dual-write drift [ADR-0002](0002-derived-not-stored.md) exists to forbid. The roster badge ("quedan 3"), the *lleno* / *waitlist* gate, and the member-app availability count are all projections over `reservation` + `class_session` — **realized as a VIEW**, consistent with the member-stats posture §4 already takes ("a VIEW, not a base table"). "Active" here reads as an un-cancelled reservation that still holds the spot (`reservada` / `asistida` occupy it; `cancelada` and `no_show` do not) — an implementation reading of §5.1's *"active reservations,"* not an additional locked rule.

### 4. A booking CONSUMES a class from the member's plan; the consume is bound to the reservation (§5.5)

The decrement is bound to **reservation creation**, not to attendance actually occurring — the class-scheduling extension of [ADR-0004](0004-saldo-stored-running-balance.md)'s stored saldo and [ADR-0005](0005-atomic-write-rpcs.md)'s atomic seam, carried to the member tier:

- **The decrement fires atomically with the reservation insert.** Invariant 5's *"booking/attendance decrements"* resolves to *reservation-creation decrements*: for a member booking ahead, that instant is the booking; for an operator-created **walk-in** (`is_walk_in`) the reservation is created at the door during *"Pasar lista,"* so the consume fires then. Either path, one reservation = one class consumed.
- **`Ilimitado` is EXEMPT.** A plan with `class_quota = NULL` / `is_unlimited = true` **never decrements** — the direct heir of the `clases_restantes IS NULL` ilimitado guard already living in `toggle_pase` / `consumirClase` ([ADR-0004](0004-saldo-stored-running-balance.md), [ADR-0005](0005-atomic-write-rpcs.md)).
- **A `no_show` still consumes.** Resolving a reservation to `no_show` **keeps the consumed class** — the member booked the spot and forfeited it; there is no refund. This is *why* the consume binds to the reservation and not to attendance.
- **`Pasar lista` does not re-consume.** Marking present transitions `reservada → asistida` and writes the `asistencias` row in one atomic write (§5); the class was already consumed at reservation creation, so `asistida` changes status, **not** balance.
- **Cancelling before the session refunds — finite plans only.** A `cancelada` on a still-`reservada` booking refunds the class, the direct sibling of `toggle_pase`'s refund guard ([ADR-0005](0005-atomic-write-rpcs.md): refund iff the reservation actually consumed *and* the plan is finite). `Ilimitado` has nothing to refund.
- **The write is ATOMIC.** The reservation insert + `subscription` decrement (and the cancel + refund) are **one all-or-nothing write** on the [ADR-0005](0005-atomic-write-rpcs.md) RPC seam (SECURITY INVOKER, `SET search_path TO ''`, EXECUTE to `authenticated` only), the member-tier sibling of `toggle_pase`. The balance math (the guarded `−1`, the ilimitado exemption, the refund guard) is the tested-TS `consumirClase` *spec* in `src/domain`; the RPC carries the transaction. There is no client-side `quedan--`.

### 5. `reservation` is the booking row; `asistencia` records the `asistida` state

A booking is **`reservation`** (`id`, `gym_id`, `class_session_id`, `member_id`, `status ∈ reservada|cancelada|asistida|no_show`, `is_walk_in`, `checked_at`, `created_at`, `cancelled_at`) with **`UNIQUE (member_id, class_session_id)`** — a member holds at most one reservation per session (re-booking a cancelled slot reuses the row). `is_walk_in` covers the operator-created walk-in (operator-writable per the member-owned/transactional RLS class, §3) — created at the door, where it consumes (§4).

*"Pasar lista"* (the pase de lista) is the operator marking attendance. It writes the existing **`asistencias`**, which gains a **`class_session_id`** (and **`reservation_id`**) FK — *"asistencias enviadas a ASIST"*. An `asistencias` row **is** the `asistida` state of its `reservation`; marking present transitions the reservation `reservada → asistida` and writes the attendance row in one atomic write. It **does not re-consume** — the class was consumed when the reservation was created (§4). Member stats (`clases_tomadas`, `racha_sem`, `favorita`, monthly usage by type) are a **VIEW** over `reservation` + `class_session`, never a base table.

### 6. `plan` = public catalog; `subscription` = live member state

The paquete model splits into catalog vs. live state, mirroring how the operator-side paquete already separates the immutable snapshot from the running balance ([ADR-0004](0004-saldo-stored-running-balance.md)):

- **`plan`** (evolves from `paquetes`; the public, member-facing catalog) — `gym_id`, `code (suelta|ocho|abierta)`, `name`, `subtitle`, `price_cents`, `cadence`, `class_quota int NULL` (**NULL = unlimited**), `is_unlimited`, `badge`, `is_featured`, with child `plan_feature`. Curated/showcased RLS (operator writes, members + anon read — it is marketing surface, §3).
- **`subscription`** (live state) — `member_id`, `plan_id`, `status (active|frozen|cancelled)`, `classes_used`, `classes_total`, `renews_on`, `period_start/end`, `frozen_until`. Member-owned/transactional RLS. Freeze is capped at **15 días/mes**. Per §6, **payment processing is deferred** — `membresía`/checkout is UI-only for v1; `subscription` tracks plan selection + status, not real charges.

Everything in §1–§6 is **gym-scoped with RLS** per [ADR-0008](0008-platform-multitenant-gym-rls-brand-modules.md); isolation is **RLS-by-membership, never the `proxy.ts` tenant header** (§5.6). New `public` tables auto-enable RLS via the existing `rls_auto_enable` event trigger — it stays on (§5.2).

## Consequences

- **Time is sortable, gateable, and timezone-honest.** `starts_at timestamptz` gives one canonical instant to order the Agenda, gate booking ("ya empezó"), stamp attendance, and survive a holiday move or a *clase especial* — none of which a weekday+string grid can express. The cost is that the calendar is **materialized rows, not a live recurrence rule**: `schedule_template` generates `class_session` rows ahead, and editing a template touches only sessions materialized after it (a generator backfill horizon, not an infinite virtual series). A future reader must not "optimize" this back into a weekday grid or a read-time recurrence expansion — that re-opens every gap §5.3 closes.
- **Occupancy can never drift.** Deriving `spots` from `count(active reservations)` makes the roster badge, the *lleno* gate, and the member availability count one source of truth — there is no `spots` field to desync ([ADR-0002](0002-derived-not-stored.md)). The cost is a `count(...)` per occupancy read, absorbed by the VIEW. **Re-introducing a stored `spots`/`occupancy` column is forbidden** — it is the exact mock shortcut this ADR exists to refuse.
- **The plan-consume reuses the proven money seam.** A booking decrements `subscription` through the same atomic SECURITY INVOKER RPC pattern as `toggle_pase` ([ADR-0005](0005-atomic-write-rpcs.md)), with the balance math owned as the tested-TS `consumirClase` *spec* ([ADR-0004](0004-saldo-stored-running-balance.md)). reservation insert + decrement (and cancel + refund) are all-or-nothing; no half-booked, half-charged state. Because the consume binds to **reservation creation**, a walk-in created at *"Pasar lista"* consumes then, and *"Pasar lista"* on an existing reservation only flips status — no second decrement. Three load-bearing rules a future reader must not "simplify": **`Ilimitado` (`class_quota = NULL` / `is_unlimited`) never decrements**; **a `no_show` on a finite plan still consumes** (no refund); **cancelling before the session refunds on finite plans only** (the `toggle_pase` refund-guard sibling). Dropping any one silently changes what members are charged.
- **One reservation per member per session** (`UNIQUE (member_id, class_session_id)`) makes double-booking a DB invariant, not a UI hope; a re-book reuses the row rather than stacking duplicates — and the occupancy `count(active)` therefore can't be inflated by a member's own churn.
- **Multi-coach is structural, not cosmetic.** `class_session_coach` lets a session carry two coaches (or a substitute) with no schema change; **no single coach column may be added back** (§5.4). `coach`, `class_type`, `plan` are curated/showcased — operator writes, members read — so the client app stays **fully operator-driven** with no hardcoded catalog.
- **`asistencias` is the bridge, not a parallel truth.** Adding `class_session_id` + `reservation_id` FKs keeps *"Pasar lista"* writing the existing attendance table while making it the `asistida` projection of a `reservation`; member stats stay a VIEW over the ledger. The operator's existing pase-de-lista flow extends onto the scheduled-class spine rather than forking from it.
- **Deferred by design (not omitted):** payment processing is UI-only for v1 (§6); a gym still has a single nullable `room` (§6 default) — `class_session.room_id` is nullable and a second venue is a later revisit, not a gap. These are stated §6 defaults, not open questions. **Scope boundary:** this ADR does not own tenancy ([ADR-0008](0008-platform-multitenant-gym-rls-brand-modules.md)) or member identity/claim-by-match ([ADR-0009](0009-identity-two-tier-auth-member-claim.md)); it consumes `gym_id`-scoped RLS and the `member` row from them.
