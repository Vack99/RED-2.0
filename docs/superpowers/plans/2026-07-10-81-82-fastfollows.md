## Global Constraints

These bind every task. Copy them into every reviewer dispatch.

1. **The #80 written-row rule:** *an RPC's return value is not its contract; the rows it writes
   are.* Every SQL suite asserts the written rows (columns the write sets: email/gym_id/
   consent-stamps/balances/clases_restantes/consumio, and membership/attendance rows upserted) —
   never only the return value or which row was touched.
2. **The per-gym suite idiom** — template is `supabase/tests/paquete_marketing_rules.sql`:
   synthetic gym + operator (ZERO production UUIDs), operator resolved from
   `gym_membership (user_id, role in ('owner','operator'))`, `set_config('t.…')` fixtures,
   per-vector jwt-claims + `set local role authenticated` / `reset role`, nested-BEGIN
   savepoints for expected-failure vectors, written-row readbacks, exactly one outer
   `BEGIN…ROLLBACK`. Minimal `auth.users` insert shape: `supabase/tests/reservar_clase_rules.sql:49-54`.
3. **Count only what you seeded:** allow-count asserts must be scoped to the suite's own
   seeded rows (a shared scratch DB has operator-seed rows the suite didn't create);
   deny-counts stay unscoped — any visible row is a leak.
4. **Read the CURRENT function body before asserting:** grep `supabase/migrations/` for the
   LATEST `create or replace function public.<name>` / `create function` — later timestamps
   override earlier. Do not assert behavior from an old body or from this plan's summary.
5. **Never touch live.** Live prod is `hjppxawglmukfvsgmcog`; the Supabase MCP in `.mcp.json`
   is bound to it — do NOT use `mcp__supabase__apply_migration`/`execute_sql` for this work.
   Implementers do not run SQL against any database: suites are validated by the dedicated
   scratch gate task later. Your verification = careful cross-reading against function bodies
   + `pnpm lint && pnpm typecheck && pnpm test` (the pre-commit hook runs these anyway).
6. **Migration hygiene:** use the exact timestamp assigned in your task. Prefer
   `CREATE OR REPLACE` (keeps grants); any `DROP`+`CREATE` of a function MUST re-assert the
   EXECUTE lockdown block (`revoke execute … from public, anon; grant execute … to
   authenticated;` — idiom: `supabase/migrations/20260706180100_pasar_lista_sesion.sql:132-134`).
7. **Do NOT edit `supabase/tests/run-denial-suite.mjs` or `supabase/tests/rpc-coverage.json`.**
   The controller does that wiring in one pass after all suites are written (prevents
   cross-task conflicts). Rewritten files stay listed in `QUARANTINE` until then — the drift
   guard passes.
8. **Keep it lean:** every assert, vector, and helper earns its keep. No speculative
   abstraction, no defensive vectors for impossible states, no duplicating what another
   running suite already asserts. User-facing copy is Spanish, matching existing messages.
