## Headline

**10 findings — all D2 (missing pagination on growable tables) — 1 HIGH + 9 MED**, across 3 DAL modules. Dominant pattern: unbounded `.select()` on growable tables (`asistencias`, `clientes`, `ventas`) with no `.limit()`/`.range()`, silently capped at ~1000 rows by PostgREST — a **cost AND silent-data-loss** risk as the gym's history grows. Several reads pull whole tables only to aggregate (counts/sums) in JS, where DB-side aggregation is the right fix. No N+1 (D1) or render-fanout (D3) findings in the audited areas.

> Most MED findings carry a bounding predicate (per-client or per-date), so their real risk hinges on the single-gym business ceiling. The single-tenant cardinality lens was **intentionally not added** (a future multi-tenant clone will use a different DB). Deferred this run: `ventas.ts`, `cobro.ts` (MED) + 11 low-priority areas.

## Findings (priority order, blast-radius descending)

### F-001 — `getMarcadas` reads the ENTIRE `asistencias` table — **HIGH** → ✅ FIXED (`d958423`)
- **src/lib/data/asistencia.ts:20:7** · D2
```
const { data } = await supabase
  .from("asistencias")
  .select("fecha, cliente_id")
  .is("deleted_at", null);
...
for (const row of data) { (map[row.fecha] ??= []).push(row.cliente_id); }
```
- **Why:** the only filter is `.is(deleted_at, null)` (soft-delete — does NOT bound cardinality). No temporal/identity predicate, no limit; the whole table is iterated into a per-day map. Worst blast radius: unbounded **and** un-scoped.
- **Fix (as shipped):** paginate the full read via a `.range()` loop (`.order("fecha")` for stable windows), not a date window — see Remediation outcome above for why full-history was the correct shape given the calendar's any-past-month browsing.
