# Member Registration Phase 1 — Gap Audit Findings

**Date:** 2026-07-08 · **Scope:** downstream gaps in the shipped member-registration Phase 1 · **Method:** 5-dimension adversarial workflow (opus-4.8; investigate → per-finding skeptic verify), 22 findings, 19 confirmed gaps + 3 working-as-intended. Analysis only — no code or DB was changed.

## Verified landing (live production facts)

- Both Phase 1 migrations are applied live: `reclamar_create_zero_saldo` (`20260707030000`) and `registrar_venta_capture_email` (`20260707031000`).
- `registrar_venta`'s live signature is the 12-arg version with `p_email`; the old 11-arg overload was dropped.
- `reclamar_o_crear_cliente`'s create-path sets `clases_restantes = 0` (finite), not NULL.
- **Live DB:** 39 `clientes`, **0** with `auth_user_id` (nobody has self-registered yet), **8** with an email → ~31 rows have no email at all.
- Every gap below is therefore currently **LATENT** but **structurally certain** the first time self-registration is used against the existing roster.

## The root defect

Email is the ONLY join key between the two doors that create a member — the admin sale (`registrar_venta`, Door 1) and client self-registration (`reclamar_o_crear_cliente`, Door 2). That key is optional, never `.email()`-validated, and compared exact-string / case-sensitive (`lower(email) = lower(v_email)`, no canonicalization). There is no backfill, no merge, no dedup anywhere in the codebase. The consequence is duplicate `clientes` rows plus orphaned paid balances the member's app can never see (RLS's `clientes_member_select` requires `auth_user_id = auth.uid()`, so it hides the unclaimed `auth_user_id IS NULL` paid row) — recoverable only by a manual DB edit or by double-charging the member.

| Trigger | Frequency | Result |
|---|---|---|
| No email captured on the sale | 31/39 live rows — **the default** | Self-register mints a fresh `clases_restantes=0` row; paid balance orphaned on the emailless row |
| Member self-registers with a different email than the one captured | Common (people use a different everyday address) | Same split: duplicate empty row, paid row stays unclaimed |
| Typo / Gmail dot or `+alias` vs the captured email | Reachable even when the operator "wired it up" | `lower()` folds case only — no canonicalization → v_n=0 → duplicate |
| Two same-email unclaimed rows (`v_n >= 2`) | Operator data-entry error (two NUEVO sales) | `if v_n = 1` false → create path → third duplicate, two orphaned paid rows |
| Operator sells via NUEVO to an already self-registered member | Whenever the operator misses the existing row | Package lands on an `auth_user_id NULL` orphan row the member's app can never read |
| Once a duplicate exists | — | No merge/relink tool anywhere → recovery is manual DB surgery |

## Confirmed gaps — detail

### Q1 — Walk-in adds email later

> Phase 1 gave the admin sale one new capability: `registrar_venta` gained a 12th nullable param `p_email` that stores `clientes.email` ONLY on the NEW-cliente INSERT path, surfaced by an optional, unvalidated email input in the Vender > NUEVO form. Self-registration (Door 2) runs `reclamar_o_crear_cliente`, which matches a caller's VERIFIED email against UNCLAIMED `clientes` rows (`auth_user_id IS NULL, lower(email)=lower(mine)`) and claims only on EXACTLY one match, else mints a fresh `clases_restantes=0` row. Across all 25 migrations, `clientes.email` is written in exactly one statement (the NEW-cliente sale INSERT) and is immutable thereafter — no email-edit UI, no email-update RPC, no merge/relink path anywhere.

#### No code path can add or edit email on an existing `clientes` row after the sale — `high` (CONFIRMED)

- **Scenario:** Operator sells a package to a walk-in with NO email (cash-only). Later the person asks to use the online app, so the operator wants to attach their email to the existing paid row — via the client's ficha / edit sheet, and/or by re-selling to them as an EXISTING client.
- **What happens today:** `clientes.email` is written in exactly one statement across the whole system — the NEW-cliente INSERT inside `registrar_venta` (`20260707031000_registrar_venta_capture_email.sql:42-43`). No UPDATE anywhere sets email. An operator therefore has no in-app way to attach or correct a member's email on an existing row: the edit sheet exposes only nombre/tel, `actualizar_cliente` ignores email, and re-selling as EXISTING runs the `registrar_venta` UPDATE branch that never touches email. Because the verified-email claim key can never be repaired, a walk-in sold a cash package with no/wrong email who later self-registers falls to v_n=0 and gets a fresh duplicate row (`clases_restantes=0`), stranding the paid balance on the old, now-unlinkable row. Nuance: the booking symptom has an operational workaround (re-sell classes onto the new auth-linked row), but the original paid balance and the email itself remain permanently uncorrectable in-app — only a direct DB write could fix them. The precondition is common: email capture on sale is optional and most existing rows have no email.
- **Consequence:** An operator can never backfill or correct a member's email. Once a `clientes` row exists without a (correct) email, the verified-email claim key is permanently unfixable, which directly enables the orphaned-balance duplicate below. This is the root enabling gap.
- **Suggested direction (analysis only):** A fix would need an email-write surface on the existing row (extend `actualizar_cliente` + the edit sheet + DAL schema to include email, or a dedicated set-email RPC), plus a decision on validation/uniqueness of email at that write.

#### Walk-in sold without email who self-registers gets a fresh 0-balance duplicate; paid classes orphaned and unbookable — `high` (CONFIRMED)

- **Scenario:** A paying member has a `clientes` row created by an operator sale with `clases_restantes>0` and email NULL (39 live clientes, 8 with email — so ~31 rows are in exactly this state). The person self-registers in the client app (Door 2) with their real email `juan@gmail.com`, verifies, and `/auth/confirm` invokes `reclamarCliente(gymId)`.
- **What happens today:** For an operator-created (Door 1) cliente with `clases_restantes>0`, `auth_user_id NULL`, and email NULL, when the same person self-registers with a real verified email, `reclamar_o_crear_cliente` counts unclaimed rows by `lower(email)=lower(v_email)`; the NULL email is not counted (v_n=0), the idempotency check (`auth_user_id=me`) misses the NULL-auth paid row, and the RPC falls through to INSERT a brand-new cliente owned by the auth user with `clases_restantes=0` (migration `20260707030000` lines 47-48, 38-45, 69-77). Phone/tel on the paid row is ignored as a claim key. No unique constraint blocks the second row (only `clientes_auth_user_id_per_gym` exists; NULL-auth rows are explicitly unconstrained), so a true duplicate is created and the paid row is left orphaned (`auth_user_id` and email still NULL). The member's new account has `clases_restantes=0`, so `reservar_clase` rejects every booking with "Sin clases disponibles" (`20260706170000` lines 179-181, decrement guard 221-227). The purchased classes remain on the orphan row, visible as a second roster entry, reachable by no online action; there is no in-app merge/relink and `actualizar_cliente` cannot even move the email, so recovery requires manual operator/DB intervention. Scoping nuance: the gap triggers whenever the paid row's email does not case-insensitively equal the verified self-register email — NULL email (~31 of 39 rows) is the primary instance, but a stored-but-mismatched email hits the identical fall-through; a matching email would correctly claim and carry the balance.
- **Consequence:** The member's new online account shows 0 classes and `reservar_clase` rejects every booking with "Sin clases disponibles", even though they already paid. Their purchased classes sit on a now-duplicate, orphaned `clientes` row (still visible in the admin roster as a second entry for the same person), reachable by no online action.
- **Suggested direction (analysis only):** Depends on the no-email-backfill fix (capture/correct the email BEFORE self-register so the claim matches) and/or a phone-based reconciliation or an operator-driven claim/merge. Would need to decide the match key and how to handle the already-minted duplicate.

#### No merge/re-link path exists, and the claim's idempotency short-circuit forecloses any later recovery — `high` (CONFIRMED)

- **What happens today:** Once a member self-registers and the claim mints a fresh 0-balance duplicate (because the paid admin-created row has an absent or mismatched email vs the verified signup email), the paid classes on the orphan row are unrecoverable through any in-app path: (a) no merge/relink/transfer/reassign RPC or UI exists anywhere; (b) `auth_user_id` can never be re-pointed by any in-app write, so the orphan cannot be re-owned; (c) there is no in-app way to even backfill email onto an existing cliente (`actualizar_cliente` and the admin edit sheet touch only nombre+tel), so the scenario's "future email backfill" itself needs direct DB access; and (d) even after such a manual email backfill, the claim's idempotency short-circuit (`20260707030000:41-44` returns the caller-owned duplicate and `return`s before the email-match block) guarantees a re-run never re-links. The only in-app "recovery" is `registrar_venta` re-selling classes onto the member's duplicate row — a new venta/double-charge that OVERWRITES rather than transfers the balance. True recovery requires a manual DB edit (repoint `auth_user_id` or move `clases_restantes`). Precondition is reachable and common: email on the sale is optional and never validated (a typo forwards verbatim), and only 8/39 live clientes have any email; it is bounded only by the fact that Door 2 self-registration has not yet been used in production (0/39 rows have `auth_user_id`).
- **Consequence:** The orphaned paid classes are unrecoverable through any in-app path: they cannot be transferred to the member's account, and the one-shot claim guarantees a second attempt can never re-link them. Recovery requires a manual DB edit or re-selling (double-charging) the member. The paid balance is effectively stranded.
